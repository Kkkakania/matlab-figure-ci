# 规则设计说明

`matlab-figure-ci` 不是版权清洗工具。

它的目标是让公开 MATLAB 绘图仓库在发布前更容易发现四类风险：隐私残留、来源不清、风险二进制文件、gallery 输出缺失。工具发现风险以后，维护者应该回到材料本身确认来源、授权和公开边界，而不是把提示词、作者行或版权行删掉就继续发布。

## 规则层级

当前默认规则分成三类：

| 类型 | 默认严重级别 | 含义 |
|---|---|---|
| 隐私规则 | error | 可能包含邮箱、本地路径或个人信息关键词，默认会导致 `scan` / `check` 失败。 |
| 来源规则 | warning | 可能包含作者、版权、许可证或平台来源线索，默认提醒复核，不直接导致 CI 失败。 |
| 扩展名规则 | error / warning | 对 `.fig`、`.mat`、`.p` 等风险文件报 error，对 `.pdf`、`.mlx`、`.zip` 等报 warning。 |

默认策略故意偏保守：它不会判断一个文件一定侵权，也不会判断一个文件一定安全。它只是把公开仓库中最容易被忽略的风险提前暴露出来。

## 隐私规则

隐私规则默认开启，并且默认是 error。常见命中包括：

- 邮箱形式的字符串。
- 本地绝对路径，例如 macOS、Linux、Windows、WSL、Codespaces 和外接盘路径。
- 个人证件、支付卡、联系电话、住址和社交账号等中文敏感信息线索。

所有隐私命中都会在终端、Markdown 报告和 JSON 报告中脱敏为 `<redacted>`。报告只保留相对路径、行号、规则 ID 和严重级别，避免质量工具自己泄露隐私内容。

## 来源规则

来源规则默认是 warning，不默认让 CI 失败。常见命中包括作者标记、版权标记、第三方许可证标记、二次发布标记和中文内容平台痕迹。

这些规则的目的不是鼓励删除来源信息。相反，它们提醒维护者停下来确认：

- 这个文件是不是本仓库原创重写的？
- 是否复制了第三方源码、示例、课程材料、论坛内容或论文图？
- 如果它来自第三方，许可证是否允许进入这个仓库？
- 是否应该改成 clean-room 代码、合成数据或链接到外部来源，而不是把材料提交进仓库？

如果一个项目希望 release gate 更严格，可以在命令行使用：

```bash
mfigci check --config mfigci.yml --fail-on-warnings
```

也可以在配置中开启：

```yaml
strict:
  fail_on_warnings: true
```

## 扩展名规则

默认 error：

- `.p`
- `.mat`
- `.fig`
- `.doc`
- `.docx`
- `.xlsx`
- `.vsd`

这些文件很容易携带不可审查内容、历史残留、个人信息、第三方数据或二进制状态。公开科研代码仓库通常应优先提交文本源码、合成数据生成脚本和可复现配置。

默认 warning：

- `.pdf`
- `.mlx`
- `.zip`

PDF 在科研绘图仓库里可能是合法的矢量输出，所以不能默认强制失败。但 PDF 如果出现在 `docs/`、`raw/` 或不清楚来源的目录中，仍然值得人工复核。

## Gallery PDF 为什么可以放行

启用 `matlab-figures` preset 后，`gallery/` 下的 PDF 可以作为图表输出被允许：

```yaml
presets:
  - matlab-figures

gallery:
  path: gallery
  allowed_extensions:
    - .png
    - .svg
    - .pdf
```

这个例外只适合渲染结果目录，不应该扩大成 `**/*.pdf`。如果仓库里有论文、教材、第三方图集或截图包，应该放在公开仓库之外，或者只用链接和来源说明引用。

## 什么时候该修改规则

可以修改规则的场景：

- 你的团队有额外的隐私关键词或实验室内部路径需要扫描。
- 你的 gallery 输出目录不是 `gallery/`。
- 你希望 release 分支把 provenance warning 也当作失败处理。
- 你有经过审查的本地 preset，需要在多个仓库复用。

不建议修改规则来绕过的问题：

- raw 素材包已经进入仓库。
- `.mat`、`.fig`、`.p` 文件来源无法说明。
- 报告命中第三方作者、平台、许可证线索，但没有完成授权确认。
- 为了让 CI 变绿而把大目录加入 `scan.exclude`。

`mfigci` 的价值不在于把所有 warning 消掉，而在于让维护者知道哪些内容已经确认、哪些内容还没有确认。
